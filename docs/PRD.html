1. Problem & Vision

Everyday email + calendar work costs too many clicks across two apps. Turning a "let's meet Thursday" email into a sent invite is 6+ clicks. The inbox buries what actually needs a reply. Search is slow and keyword-only. Nothing is keyboard-driven.

Vision: SlotNest is a single, keyboard-first surface where you triage, reply, search, and schedule at the speed of thought — with an AI agent that executes real Gmail/Calendar actions from one sentence. It does not reimagine Gmail's look; it removes the friction Gmail and Calendar leave behind.

North star: Solve real workflow pain — not a clone, not AI-for-AI's-sake. Every feature must measurably remove clicks or surface what matters.

2. Goals & Non-Goals

Goals

  • Real Gmail + Google Calendar integration through Corsair (mandatory; non-hardcoded).
  • At least one obviously-better-than-Gmail workflow, demoable in <30s.
  • Keyboard-first UX with a unified command bar.
  • AI that earns its place: triage, drafting, scheduling, agent execution.
  • Deployed live link + YC-style demo video + open-source repo.

Non-Goals (v1)

  • Multiple Gmail/Calendar accounts per user (one of each in v1).
  • Mobile-native app; full Gmail label/filter management parity.
  • Team/shared inboxes, billing, or org admin.
  • Providers beyond Gmail + Google Calendar.

3. Users & Scenarios

Primary user: a busy professional who lives in Gmail + Calendar and wants to move faster with the keyboard.

Signature scenarios

  • One-sentence scheduling: ⌘K → "Send a calendar invite to friend@corsair.dev at 9 AM next Thursday and email him I look forward to it." → done.
  • Triage in seconds: open inbox sorted by priority; "3 need a reply" — clear them with keyboard, each with a pre-written draft waiting.
  • Email → invite: on a meeting email, one keystroke creates & sends an invite to the sender at an extracted time.
  • Instant search: sub-second semantic + keyword search over locally cached mail.

5. Architecture & Key Decisions

Auth

better-auth (existing). Corsair tenant id = session.user.id.

Integrations

Corsair multi-tenant OAuth; corsair.withTenant(userId).gmail / .googlecalendar.

Data freshness

Corsair webhooks → Postgres local cache. ngrok locally, deploy URL in prod.

Search

Hybrid: pgvector embeddings (OpenAI text-embedding-3-small) + Postgres full-text.

Agent

OpenAI Agents SDK + OpenAIAgentsProvider from @corsair-dev/mcp (per docs/corsair/openai-agents.md).

UX shell

Unified cmdk command bar routes commands vs natural language.

API layer

tRPC routers (existing gmail, connections) over Drizzle/Postgres.

Deploy

Vercel + hosted Postgres (Neon/Supabase) with pgvector.
Docs rule: for any Corsair API, read the matching file in docs/corsair/ first. If a needed area (e.g. an MCP adapter detail) isn't covered, add the doc before coding — do not guess the API.

6. Triage Model (two-axis)

Each incoming email is classified on ingest by a cheap LLM over subject + snippet, then stored in Postgres and combined into a sortable priority.

AxisValuesPurpose
ActionNeeds reply · FYI · IgnoreWhat the user must do. Drives the triage view + auto-draft trigger.
UrgencyUrgent · Normal · LowHow time-sensitive. Drives sort order within actions.

Combined priority sort: Needs-reply+Urgent → … → Ignore+Low. "Needs reply" emails auto-trigger a draft reply (Signature #1).

7. Step-by-Step Build Plan

Strict dependency order. If the deadline hits mid-way, you still have a working keyboard-driven client on real data.

Phase 0 — Foundation P0 · Day 1

  • Land plan 001: Corsair OAuth connect for Gmail + Calendar (connect/callback routes, connections page).
  • Verify corsair.withTenant(userId) returns real data for both plugins.
  • Confirm Postgres + Drizzle schema, hosted DB, pgvector extension enabled.

Phase 1 — Real data flowing P0 · Day 1–2

  • Set up Corsair webhooks (Gmail + Calendar) → persist to Postgres local cache. ngrok for local.
  • Inbox read over cached data; thread/email open view.

Phase 2 — Keyboard shell & write actions P0 · Day 2–3

  • Unified cmdk command bar; core shortcuts (navigate, compose, reply, search).
  • Email send / reply / draft. Calendar create + send invite.

Phase 3 — Agent P0 · Day 3–4

  • Wire OpenAI Agents SDK + Corsair MCP provider; route NL from command bar to agent.
  • Validate the hero command: invite + email in one sentence.

Phase 4 — Intelligence & speed P1 · Day 4–5

  • Two-axis triage on ingest; priority inbox view.
  • Hybrid search (embed on ingest + full-text).
  • Email→invite one-shot.
  • Signature #1: auto-draft replies. Signature #2: NL free-slot scheduling.

Phase 5 — Stretch + ship P2 · Day 5–6

  • If time: daily digest, thread summary, modify/cancel events.
  • Deploy to Vercel; production webhooks; polish.
  • Record YC-style demo video; README + Corsair-features + bonus list; X/LinkedIn posts.

10. Risks & Mitigations

RiskMitigation
Over-scope: too many features, all half-done.Strict P0→P1→P2 order; signature features capped at 2; digest/summary parked.
Agent/MCP wiring unfamiliar.Follow docs/corsair/openai-agents.md exactly; build agent only after read/write work standalone.
Webhooks flaky locally.ngrok for dev; fall back to on-demand fetch for a view if a webhook lags (don't block UI).
Sending real email/invites in demo.Use a dedicated test recipient (e.g. @corsair.dev); confirm-before-send for destructive agent actions.
Deploy/webhook config eats the last day.Deploy by Phase 4, not Phase 5; verify prod webhooks early.
LLM latency on triage at ingest.Cheap/fast model; score async on ingest, not on read path.
